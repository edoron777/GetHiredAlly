INTRODUCTION — TextStylerService Phase 1: Database Schema

PROJECT CONTEXT:
We are building a TextStylerService for GetHiredAlly that will provide
rich text formatting preservation when importing Word/PDF files.

This is Phase 1 of 6 phases. Phase 1 adds database columns to store
HTML-formatted content alongside plain text.

WHAT WE'RE ADDING
═══════════════════════════════════════════════════════════════

Table: user_cvs
┌─────────────────────────────────────────────────────────────┐
│  EXISTING COLUMNS (no changes)                              │
│  - id, user_id, filename, content, etc.                     │
├─────────────────────────────────────────────────────────────┤

Table: cv_scan_results
┌─────────────────────────────────────────────────────────────┐

═══════════════════════════════════════════════════════════════
WHY HTML STORAGE?
═══════════════════════════════════════════════════════════════

Current Problem:
- Word/PDF files converted to plain text with [MARKERS]
- [BOLD], [H1], [BULLET] markers are inconsistently handled
- Rich formatting (italic, links, tables) is lost

Solution:
- Store HTML version alongside plain text
- Plain text used for: AI analysis, text search, scoring
- HTML used for: User display with full formatting
